# 배포 트러블슈팅 로그 (jeongyeol-finder)

> 빌드·배포 중 만난 오류를 기록한다. **다음에 빌드/배포하기 전 이 문서를 먼저 읽고**, 같은 함정을 반복하지 않는다.
> 새 오류가 생기면 "증상 / 원인 / 해결 / 재발방지" 형식으로 아래에 추가한다.

## 환경
- Next.js **16.2.7** (App Router) · React 18.3.1
- Cloudflare 배포: `@opennextjs/cloudflare` 1.19.x + `wrangler` 4.x (Workers, `nodejs_compat`)
- Vercel 배포 **병행 유지** (Cloudflare 전용 설정은 `CF_BUILD=1`로 분기 → Vercel 빌드 무영향)
- OS: Windows 11 (OpenNext는 Windows 공식 비권장 — 가능하면 WSL)

## 정상 경로 (요약)
- 로컬/Vercel: `npm run build` (Turbopack), `npm test`
- Cloudflare 빌드: `npm run cf:build` → 미리보기 `npm run cf:preview` → 배포 `npm run cf:deploy`
- `cf:build`는 `scripts/cf-build.mjs`가 ① 표 TS 생성 ② `next build --webpack` ③ `opennextjs-cloudflare build --skipNextBuild` 순서로 실행
- **표 데이터 수정 시:** `data/전결_검색테이블_통합.csv` 고친 뒤 `npm run data:build` (빌드 스크립트도 매번 자동 재생성)

## 오류 기록

### 1. 제미나이 키를 넣어도 동작 안 함 ("결과 생성 실패")
- **증상:** 유효한 Gemini 키 입력 후에도 LLM 응답 실패. 폰에서 "LLM 필요"/"결과 생성 실패".
- **원인:** 기본 모델 `gemini-1.5-flash`가 **폐기된 모델** → 404. `route.ts` 에러 분기(401/quota/429) 어디에도 안 걸려 마지막 502 generic 메시지로 떨어짐.
- **해결:** `lib/llm.ts` 기본 모델 → `gemini-2.5-flash`. `route.ts`에 404/모델 미지원 전용 분기 추가(키 문제 아님 안내). `.env.example` 갱신.
- **재발방지:** LLM 모델명은 **현행 모델인지 주기적 확인**. Vercel/Cloudflare 환경변수에 `GEMINI_MODEL`이 옛 값으로 박혀 있으면 코드 기본값을 덮어쓰니 거기도 점검.

### 2. `@opennextjs/cloudflare` 최신판이 Next 15+ 요구 (peer 충돌)
- **증상:** `npm install @opennextjs/cloudflare@latest` 시 ERESOLVE — peer `next>=15.5.18`인데 프로젝트는 Next 14.
- **원인:** 어댑터 1.x가 Next 15+ 전용.
- **해결:** Next 16으로 업그레이드(이 앱은 `cookies()/headers()/params` 미사용 + API 라우트 POST·동적이라 breaking change 영향 작음).
- **재발방지:** OpenNext 어댑터 버전과 Next 버전 호환표를 먼저 확인. Next 14 유지가 필요했다면 어댑터 0.6.x 핀이 대안.

### 3. Turbopack 청크 로드 실패 (`ChunkLoadError`)
- **증상:** OpenNext 미리보기에서 GET `/`까지 **500**. 로그: `Failed to load chunk server/chunks/ssr/[root-of-the-server]__...._.js ... Not found`.
- **원인:** Next 16 `next build`가 **Turbopack 기본**. OpenNext가 Turbopack 서버 청크 출력을 안정적으로 못 읽음.
- **해결:** Cloudflare 빌드는 **webpack 강제** — `next build --webpack` 후 `opennextjs-cloudflare build --skipNextBuild`. (OpenNext는 `.next/server/webpack-runtime.js` 유무로 번들러 감지)
- **재발방지:** Cloudflare용 빌드는 항상 webpack. `cf-build.mjs`에 고정돼 있으니 우회하지 말 것. Vercel은 Turbopack 그대로 둠.

### 4. standalone 출력 누락 (`pages-manifest.json` ENOENT)
- **증상:** `--skipNextBuild` 후 `.next/standalone/.next/server/pages-manifest.json` 없음.
- **원인:** 직접 돌린 `next build`에 `output:"standalone"`이 없어 standalone 디렉터리 미생성. (OpenNext 자체 빌드 땐 자동 주입되지만 skip 시 우리가 줘야 함)
- **해결:** `next.config.mjs`에서 **`CF_BUILD===1`일 때만** `output:"standalone"` 설정.
- **재발방지:** Cloudflare 빌드는 반드시 `CF_BUILD=1` 환경에서(=`cf-build.mjs` 경유). 직접 `next build --webpack`만 돌리면 standalone 안 나옴.

### 5. standalone 디렉터리가 프로젝트명 하위로 중첩
- **증상:** 출력이 `.open-next`가 기대하는 `.next/standalone/.next/...`가 아니라 `.next/standalone/jeonjyeol-chatbot/.next/...`.
- **원인:** Next가 상위 폴더(`C:\Users\user`)를 워크스페이스 루트로 오인(여러 lockfile 등) → 경로 한 단계 중첩.
- **해결:** `CF_BUILD`일 때 `outputFileTracingRoot`를 **프로젝트 폴더로 고정**.
- **재발방지:** standalone 경로가 또 어긋나면 제일 먼저 `outputFileTracingRoot` 의심.

### 6. (핵심) 워커에서 표가 비어 로컬 검색 전부 실패
- **증상:** 미리보기에서 GET `/`는 200인데, `"예산의 변경"`(정확 일치)조차 `needsKey` 반환. 즉 `resolveLocal`이 항상 null → 표가 비어 있음.
- **원인:** `import json from "*.json"`가 **Next 서버 빌드에서 에셋 파일로 외부화**됨. 번들엔 데이터 미포함, 별도 `.json` 파일로 복사 → **Workers엔 런타임 파일시스템이 없어** 읽지 못함. (확인법: 번들 route.js에서 `담당자` 등 표의 흔한 값 grep — 거의 0이면 외부화된 것)
- **해결:** JSON 대신 **TS 모듈로 임베드**. `scripts/gen-table.mjs`가 `lib/tableData.generated.ts`(`const rows = [...]; export default rows;`) 생성 → TS 배열 리터럴은 webpack이 외부화 못 해 항상 번들 인라인.
- **재발방지:** Workers/엣지 런타임에선 **데이터를 fs로 읽거나 `*.json` import에 의존하지 말 것**. 빌드 타임에 JS/TS 소스로 임베드. 빌드 후 `grep -c '담당자' route.js`로 인라인 검증.

### 7. `.open-next` EPERM(권한 거부)로 재빌드 실패
- **증상:** 재빌드 시 `EPERM ... .open-next`.
- **원인:** 이전 미리보기(`workerd`/`wrangler`) 프로세스가 폴더를 잠금.
- **해결:** 재빌드 전 `workerd|wrangler|opennext|miniflare` 프로세스 모두 종료 후 `.open-next`/`.next` 삭제. (workerd는 자식 프로세스가 여러 개니 전부 kill)
- **재발방지:** 미리보기를 백그라운드로 띄웠다면 **재빌드 전 반드시 종료**.
